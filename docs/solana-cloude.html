Solana Fundamentals — Runtime, Accounts, Transactions\ \
\
\
\Explain Proof of History (PoH) and how it enables Solana's performance.\ \Core\ \▾\ \ \
\\PoH is a cryptographic clock built on a Verifiable Delay Function (VDF).\ It creates a historical record proving that events occurred at specific points in time, without requiring validators to synchronize clocks externally.\ \Each hash output becomes the input for the next — creating a sequential, verifiable chain. Because VDFs are inherently sequential and cannot be parallelized, the output provably took a certain number of CPU cycles to compute.\ \\Why this matters for performance:\\ \ \Eliminates inter-validator clock-sync messages, drastically reducing network communication overhead.\ \Tower BFT can leverage PoH timestamps to finalize consensus faster with fewer rounds of voting.\ \Validators don't need to agree on time; they trust the VDF chain — enabling sub-second block times (\~400ms slots).\ \ \// PoH conceptual model (simplified) let hash \= sha256(prev\_hash \+ counter); // Each iteration is one "tick" — provably sequential // The count encodes elapsed time in CPU cycles\ \ \ \
\
\Describe Solana's account model. How does it differ from Ethereum's contract model?\ \Core\ \▾\ \ \
\In Solana, \everything is an account\. Unlike Ethereum where contract storage lives inside the contract, Solana separates \code\ from \state\:\ \ \\Program accounts\ — store executable bytecode (immutable once deployed without upgrade authority).\ \\Data accounts\ — store state (owned by a program, can be written by that program only).\ \\Wallet accounts\ — hold SOL, owned by the System Program.\ \\Token accounts\ — store token balances, owned by the SPL Token Program.\ \ \Key differences vs Ethereum:\ \
\ \\Feature\\Solana\\Ethereum\\ \\State storage\\Separate data accounts\\Inside contract storage\\ \\Code mutability\\Upgradeable (with authority)\\Immutable (proxy pattern needed)\\ \\Gas / Fee model\\Compute Units (CUs)\\Gas (EIP-1559)\\ \\Parallelism\\Sealevel — explicit read/write sets\\Single-threaded EVM\\ \\Rent\\Accounts require rent-exempt SOL deposit\\Storage costs are gas-based\\ \ \ \This separation is what enables Sealevel's parallel execution — transactions declare which accounts they access, so non-conflicting transactions run concurrently across multiple cores.\ \ \ \
\
\What is a Program Derived Address (PDA)? When and why would you use one?\ \Mid\ \▾\ \ \
\A \PDA\ is a deterministic account address derived from a program ID and seeds, but it does \not\ lie on the ed25519 elliptic curve — meaning no private key can sign for it. Only the program itself can sign via \invoke\_signed()\.\ \\Why use PDAs?\\ \ \\Program-controlled accounts\ — the program is the sole authority, enabling trustless escrow, vaults, and liquidity pools.\ \\Deterministic addressing\ — any client can re-derive the address from seeds without storing it.\ \\Namespacing\ — using seeds like \\[b"user", user\_pubkey\]\ creates per-user accounts that cannot collide.\ \ \// Deriving a PDA let (pda, bump) \= Pubkey::find\_program\_address( &\[b"vault", user.key().as\_ref()\], program\_id, ); // In Anchor — even cleaner: \#\[account( init, payer \= user, space \= 8 \+ Vault::LEN, seeds \= \[b"vault", user.key().as\_ref()\], bump )\] pub vault: Account\<'info, Vault\>,\ \\Common use-cases:\ token vaults, escrow accounts, stake pools, DAO treasuries, user profile accounts, and any account that a program — not a user — must own and control.\ \ \ \
\
\How does Sealevel enable parallel transaction execution? What are its constraints?\ \Senior\ \▾\ \ \
\Sealevel is Solana's parallel smart-contract runtime. Every transaction must \declare all accounts it will read and write\ before execution. The scheduler then:\ \ \Groups transactions by their read/write account sets.\ \Executes transactions touching disjoint accounts concurrently across CPU cores (and even GPU threads via SIMD optimizations).\ \Serializes only transactions that share a writable account (lock contention).\ \ \An additional optimization: Sealevel sorts transactions by program ID and runs the same program over multiple independent accounts in parallel — leveraging CPU SIMD instructions for further throughput gains.\ \\Design constraints / implications for developers:\\ \ \Hot accounts (e.g., a single AMM pool) become throughput bottlenecks when many transactions write to them simultaneously.\ \You must declare all accounts upfront — dynamic account discovery during execution isn't possible (unlike EVM).\ \To maximize parallelism, split large shared state into many smaller accounts (e.g., per-user accounts rather than one global account).\ \Cross-program invocations (CPI) that write to accounts block parallel execution on those accounts.\ \ \ \ \
\
\Explain rent exemption in Solana. What happens to accounts that fall below the threshold?\ \Core\ \▾\ \ \
\Accounts on Solana must hold a minimum SOL balance proportional to their data size to be \rent-exempt\. This prevents unbounded state growth by making state storage economically sustainable.\ \ \The current threshold is roughly \\~0.00203 SOL per 128 bytes\ of account data.\ \Accounts below this threshold are subject to periodic rent deduction. If a balance hits zero, the account is purged by the garbage collector.\ \Best practice: always initialize accounts with enough SOL to be rent-exempt from the start.\ \When closing an account, \return all lamports\ to the payer and zero the data — failure to do so is a security vulnerability (account resurrection attack).\ \ \// Closing an account properly in Anchor \#\[account( mut, close \= user // transfers lamports to user, zeroes data )\] pub data\_account: Account\<'info, MyData\>,\ \ \ \

\What is Gulf Stream and how does it reduce confirmation latency?\ \Mid\ \▾\ \ \
\\Gulf Stream\ is Solana's mempool-less transaction forwarding protocol. Because Solana's leader schedule is known in advance (via PoH), clients and validators can \forward transactions directly to the upcoming leader\ rather than broadcasting to the entire network.\ \\Benefits:\\ \ \Validators can pre-process and cache transactions before their slot begins — reducing slot time.\ \No mempool means no mempool-based front-running (though searchers use Jito bundles for MEV).\ \Reduces the number of unconfirmed transactions held in memory across the network.\ \Enables the low-latency confirmation (typically \~400ms for optimistic confirmation, \~13s for full finality) that Solana is known for.\ \ \\Downside:\ If you send a transaction to the wrong leader or a leader is skipped, your transaction may expire — you should use durable nonces or retry with fresh blockhashes for critical flows.\ \ \ \\<\!-- end qa-grid \--\> \

Rust Programming — Ownership, Lifetimes, Concurrency\ \

\
\Explain Rust's ownership model. Why is it particularly well-suited for Solana program development?\ \Core\ \▾\ \ \
\Rust enforces three ownership rules at \compile time\:\ \ \Every value has exactly one owner at any time.\ \When the owner goes out of scope, the value is dropped (no GC needed).\ \Borrowing (references) allows temporary access without transferring ownership.\ \ \\Why it matters for Solana:\\ \ \\No garbage collector\ — GC pauses would be catastrophic for a blockchain runtime targeting \~400ms slots.\ \\Zero-cost abstractions\ — high-level code compiles to the same machine instructions as hand-optimized C.\ \\Memory safety without a VM\ — eliminates buffer overflows, use-after-free, and null pointer dereferences at compile time. Microsoft estimates these account for \~70% of CVEs in C/C++ codebases.\ \\Fearless concurrency\ — the borrow checker prevents data races, enabling Sealevel's parallel execution model.\ \ \// Single owner — s2 gets ownership, s1 invalid let s1 \= String::from("solana"); let s2 \= s1; // s1 moved; using s1 here \= compile error // Borrowing — no transfer fn print\_len(s: \&String) { println\!("{}", s.len()); // just borrows } let s3 \= String::from("anchor"); print\_len(\&s3); // s3 still valid here\ \ \ \
\
\What are lifetimes in Rust? Describe a scenario in Solana program code where they become critical.\ \Mid\ \▾\ \ \
\\Lifetimes\ are Rust's way of tracking how long references remain valid, preventing dangling references at compile time. They're denoted with \'a\ syntax.\ \// Function returning a reference — requires lifetime annotation fn longest\<'a\>(x: &'a str, y: &'a str) \-\> &'a str { if x.len() \> y.len() { x } else { y } } // The returned reference lives at most as long as // the shorter of x and y\ \\In Solana context — the \AccountInfo\ struct:\\ \// AccountInfo borrows from the transaction context // Its lifetime 'info must not outlive the transaction pub struct AccountInfo\<'info\> { pub key: &'info Pubkey, pub lamports: Rc\\>, pub data: Rc\\>, pub owner: &'info Pubkey, // ... } // Anchor's Context struct ties all account lifetimes together pub struct Context\<'a, 'b, 'c, 'info, T\> { pub accounts: &'b mut T, pub remaining\_accounts: &'c \[AccountInfo\<'info\>\], // ... }\ \If you attempt to hold a reference to an account's data beyond the instruction's scope, the compiler will reject it — preventing a whole class of bugs where stale account data could be used after the execution context is gone.\ \ \ \
\
\How does Rust prevent data races, and how does this relate to how Solana programs handle concurrency?\ \Senior\ \▾\ \ \
\Rust's borrow checker enforces that at any given time, for any piece of data, you can have \either many immutable references OR one mutable reference — never both\. This is checked at compile time, eliminating data races without needing mutexes for most use cases.\ \In Solana's BPF runtime, programs execute single-threaded \within\ a transaction, but the Sealevel runtime runs multiple transactions \across\ threads. The account locking mechanism mirrors Rust's borrow rules at the runtime level:\ \ \Multiple transactions can \read\ the same account concurrently.\ \Only one transaction can \write\ to an account at a time (exclusive write lock).\ \ \For sharing state across instructions within a program, Solana uses \Rc\<RefCell\<T\>\>\ (single-threaded) for \AccountInfo.lamports\ and data — enforcing runtime borrow checks for mutable access within the same transaction scope.\ \ \ \
\
\Explain the difference between Box, Rc, and Arc. When would you encounter each in Solana code?\ \Mid\ \▾\ \ \
\ \\\Box\<T\>\\ — heap allocation with single ownership. Used for trait objects (\Box\<dyn Error\>\) and recursive data structures.\ \\\Rc\<T\>\\ — reference-counted heap pointer for multiple ownership in \single-threaded\ contexts. Used inside \AccountInfo\ to share lamport/data access within one instruction.\ \\\Arc\<T\>\\ — atomic reference count — thread-safe version of Rc. Used in Solana's validator/runtime internals (multi-threaded) but rarely in on-chain program code.\ \ \In on-chain Solana programs, you'll see \Rc\<RefCell\<\&'info mut u64\>\>\ for lamports — it allows the runtime to share mutable access to a single lamport balance across multiple \AccountInfo\ references within the same transaction, with runtime borrow checking via \RefCell\.\ \ \ \
\
\How does Rust's error handling with Result and Option compare to exception-based languages? How do you use it in Solana programs?\ \Core\ \▾\ \ \
\Rust has no exceptions. Errors are values returned as \Result\<T, E\>\ (recoverable) or \Option\<T\>\ (present/absent). This forces explicit error handling at every call site.\ \// Native Solana program error handling use solana\_program::program\_error::ProgramError; fn process(amount: u64, balance: u64) \-\> Result\ { balance.checked\_sub(amount) .ok\_or(ProgramError::InsufficientFunds) } // In Anchor — custom errors with \#\[error\_code\] \#\[error\_code\] pub enum MyError { \#\[msg("Amount exceeds available balance")\] InsufficientFunds, \#\[msg("Authority mismatch")\] UnauthorizedAccess, } // Returning errors in an instruction handler pub fn withdraw(ctx: Context\, amount: u64) \-\> Result\<()\> { require\!( ctx.accounts.vault.balance \>= amount, MyError::InsufficientFunds ); // ... Ok(()) }\ \Key advantages: errors are \visible in function signatures\, cannot be silently ignored (unlike exceptions), and the \?\ operator provides ergonomic early-return propagation.\ \ \ \ \

    \One of the most common take-home problems. The solution must use PDAs for the escrow vault, handle failure cases, and be idempotent.\

\// Anchor Escrow Program — Senior-Level Implementation  
use anchor\_lang::prelude::\*;  
use anchor\_spl::token::{self, Token, TokenAccount, Transfer};

declare\_id\!("Esc1...yourprogramid");

\#\[program\]  
pub mod escrow {  
    use super::\*;

    // Initializer locks Token A into escrow, expecting Token B  
    pub fn initialize(  
        ctx: Context\,  
        initializer\_amount: u64,  
        taker\_amount: u64,  
    ) \-\> Result\<()\> {  
        let escrow \= \&mut ctx.accounts.escrow\_state;  
        escrow.initializer       \= \*ctx.accounts.initializer.key;  
        escrow.initializer\_deposit\_token\_account \=  
            \*ctx.accounts.initializer\_deposit\_token\_account.to\_account\_info().key;  
        escrow.initializer\_receive\_token\_account \=  
            \*ctx.accounts.initializer\_receive\_token\_account.to\_account\_info().key;  
        escrow.initializer\_amount \= initializer\_amount;  
        escrow.taker\_amount       \= taker\_amount;  
        escrow.bump               \= ctx.bumps.escrow\_state;

        // Transfer from initializer ATA \-\> escrow vault (PDA)  
        token::transfer(  
            CpiContext::new(  
                ctx.accounts.token\_program.to\_account\_info(),  
                Transfer {  
                    from: ctx.accounts.initializer\_deposit\_token\_account  
                              .to\_account\_info(),  
                    to:   ctx.accounts.vault.to\_account\_info(),  
                    authority: ctx.accounts.initializer.to\_account\_info(),  
                },  
            ),  
            initializer\_amount,  
        )?;  
        Ok(())  
    }

    // Taker provides Token B, receives Token A from vault  
    pub fn exchange(ctx: Context\) \-\> Result\<()\> {  
        let escrow \= \&ctx.accounts.escrow\_state;  
        let seeds \= &\[  
            b"escrow".as\_ref(),  
            escrow.initializer.as\_ref(),  
            &\[escrow.bump\],  
        \];  
        let signer\_seeds \= &\[\&seeds\[..\]\];

        // Taker sends Token B to initializer  
        token::transfer(  
            CpiContext::new(  
                ctx.accounts.token\_program.to\_account\_info(),  
                Transfer {  
                    from:      ctx.accounts.taker\_deposit\_token\_account.to\_account\_info(),  
                    to:        ctx.accounts.initializer\_receive\_token\_account.to\_account\_info(),  
                    authority: ctx.accounts.taker.to\_account\_info(),  
                },  
            ),  
            escrow.taker\_amount,  
        )?;

        // PDA-signed transfer: vault Token A \-\> taker  
        token::transfer(  
            CpiContext::new\_with\_signer(  
                ctx.accounts.token\_program.to\_account\_info(),  
                Transfer {  
                    from:      ctx.accounts.vault.to\_account\_info(),  
                    to:        ctx.accounts.taker\_receive\_token\_account.to\_account\_info(),  
                    authority: ctx.accounts.escrow\_state.to\_account\_info(),  
                },  
                signer\_seeds,  
            ),  
            escrow.initializer\_amount,  
        )?;  
        Ok(())  
    }  
}

        \How does MEV work on Solana? How does it differ from Ethereum MEV?\  
        \Expert\  
        \▾\  
      \  
      \
  
        \\MEV (Maximal Extractable Value)\ on Solana differs architecturally from Ethereum:\  
        \
  
          \  
            \\Dimension\\Ethereum\\Solana\\  
            \\Mempool\\Public mempool (visible to all)\\No traditional mempool; Gulf Stream forwards to leaders\\  
            \\Ordering mechanism\\Gas price auction\\Jito bundle tips \+ priority fees\\  
            \\Primary MEV infra\\Flashbots MEV-Boost\\Jito-Solana validator client\\  
            \\Sandwich attacks\\Common via mempool watching\\Harder but possible via Jito mempool (suspended March 2024)\\  
            \\Arbitrage\\Flash loans \+ atomic swaps\\Atomic multi-hop swaps in single tx, no flash loans needed\\  
          \  
        \  
        \\Jito's role:\ By late 2024, over 90% of Solana's stake ran the Jito-Solana validator client. Jito's bundle system allows searchers to submit bundles of ordered transactions with tips, and validators select highest-tip bundles during block assembly. In Q1 2025, Jito bundles accounted for over 22% of total validator rewards.\  
        \\As a smart contract developer:\ Design protocols to be MEV-resistant. Use TWAPs (time-weighted average prices) for oracle prices, add slippage limits with on-chain checks, and consider commit-reveal schemes for sensitive operations.\  
      \  
    \

        \What is Firedancer, and how does it change the validator ecosystem?\  
        \Expert\  
        \▾\  
      \  
      \
  
        \\Firedancer\ is Jump Crypto's independent Solana validator client implementation, written in C from scratch (separate from the Rust-based Agave/Solana Labs client). Key implications:\  
        \  
          \\Client diversity\ — previously, a bug in the single Solana Labs client could halt the entire network. Firedancer provides a second independent implementation, dramatically improving resilience.\  
          \\Performance\ — Frankendancer (partial Firedancer integration) demonstrated 1 million TPS in testnet conditions in September 2024, compared to \~4,000 TPS in production today.\  
          \\Separate consensus path\ — the network can continue if one client has a bug, as long as 67% of stake runs a non-buggy client.\  
        \  
        \For program developers, Firedancer is largely transparent — programs run identically. However, it signals that the network's throughput ceiling is much higher than current production performance, so future programs should be designed for high-volume environments.\  
      \  
    \

    \
  
      \
  
        \What are Token Extensions (Token-2022)? Name 3 extensions and their use cases.\  
        \Senior\  
        \▾\  
      \  
      \
  
        \Token Extensions (the SPL Token-2022 program) is a superset of the original SPL Token program, enabling mint and account-level extensions:\  
        \
  
          \
  
            \Confidential Transfers\  
            \Uses ElGamal encryption to hide transfer amounts on-chain while preserving verifiability. Ideal for institutional DeFi and compliant private payments without full ZK overhead.\  
          \  
          \
  
            \Transfer Hooks\  
            \Invoke a custom program on every token transfer. Enables royalty enforcement, compliance blocklists, loyalty point tracking, and NFT-specific logic without wrapping tokens.\  
          \  
          \
  
            \Interest-Bearing Tokens\  
            \Accrue interest automatically without rebasing — the token's "amount" at display time is scaled by a stored interest rate. Perfect for yield-bearing stablecoin designs (like cTokens).\  
          \  
          \
  
            \Permanent Delegate\  
            \Grants a persistent authority the ability to transfer or burn any token from any holder's account — used for protocol-level compliance, subscription services, or stream payments.\  
          \  
        \  
      \  
    \

    \
  
      \
  
        \How is ZK being integrated into the Solana ecosystem? What is zkCompression?\  
        \Expert\  
        \▾\  
      \  
      \
  
        \ZK (Zero-Knowledge) is being integrated at several layers:\  
        \  
          \\Confidential Transfers\ (Token-2022) — uses ZK proofs (specifically ElGamal \+ sigma protocols) to prove transfer validity without revealing amounts.\  
          \\Light Protocol / zkCompression\ — compresses account state into a Merkle tree stored as on-chain transaction data (much cheaper than full account rent). Programs can operate on compressed accounts, proving state via ZK Merkle proofs. This dramatically reduces state storage costs — e.g., compressed NFTs (Bubblegum) reduced minting costs from \~0.01 SOL to \~0.000005 SOL per NFT.\  
          \\Solana's SVM \+ ZK rollups\ — exploratory projects aim to build ZK-rollup layers that post proofs to Solana mainnet, using it as a DA (data availability) layer.\  
          \\Groth16/PLONK verifiers\ — on-chain ZK verifier programs using Solana's precompiles for curve operations (bn254 curve operations).\  
        \

      \
  
        \Explain Solana's Turbine block propagation protocol.\  
        \Expert\  
        \▾\  
      \  
      \
  
        \Turbine solves the block propagation bandwidth problem — broadcasting a full block to thousands of validators would require enormous bandwidth from the leader. Instead:\  
        \  
          \The leader breaks blocks into small \shreds\ (\~1,280 bytes) using Reed-Solomon erasure coding.\  
          \Shreds are organized in a \binary tree of validators\ — each validator propagates shreds to its children in the tree.\  
          \Each validator only receives and forwards a portion of the block, then uses erasure codes to reconstruct the full block from partial data.\  
          \This reduces per-node bandwidth requirements from O(N) to O(√N), enabling the network to scale to thousands of validators without a bottleneck at the leader.\  
        \  
        \For developers, Turbine explains why large transactions and accounts with lots of data cost more — they require more shreds to propagate. The 1,232-byte transaction size limit directly ties to Turbine's shred size architecture.\
